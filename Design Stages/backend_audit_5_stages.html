Breakingdatabase.py uses deprecated declarative_base() import
The code imports from sqlalchemy.ext.declarative import declarative_base. This path was deprecated in SQLAlchemy 1.4 and removed in 2.x. The file pins SQLAlchemy==2.0.30 in requirements, so this will throw an ImportError immediately at startup.

Fix:
from sqlalchemy.orm import DeclarativeBase

class Base(DeclarativeBase):
    pass

Pydantic-settings v2 requires env_nested_delimiter='__' in the SettingsConfigDict to resolve nested models from environment variables. Without it, ICP.MIN_EMPLOYEES etc. can never be overridden at runtime, and any .env file setting ICP__MIN_EMPLOYEES=15 will be silently ignored.

Fix:

    env_file=".env",
    env_file_encoding="utf-8",
    env_nested_delimiter="__"   # add this
)

Bugicp_filter.py: "M&A" check uses `any(stage in funding_stage ...)` — inverted logic
The check is any(stage in funding_stage for stage in stagnant_stages) where stagnant_stages = ["Series D", "Series E", "Public", "M&A"]. This checks if the stagnant stage string is a substring of funding_stage, but the variable name stage iterates over your list items, not the input. It should be the other way: check if funding_stage (the input) matches any item in your list.

In practice both work for "Series D" since both directions of substring match return True. But it will fail silently for partial strings like "Pre-Series A" matching "Series" if the list ever grows. Use any(funding_stage.lower() == s.lower() for s in stagnant_stages) for exact matching.

Breakingpipeline_scheduler.py: raw SQL against SQLAlchemy session — wrong execution API
The code calls db.execute("SELECT intent_score FROM lead_snapshots WHERE domain=? ...", (domain,)) on a SQLAlchemy Session object. In SQLAlchemy 2.x, sessions do not accept raw strings — you must wrap them in text(). Also, the parameter placeholder for SQLite via SQLAlchemy is :domain not ?.

Fix:
from sqlalchemy import text
row = db.execute(
    text("SELECT intent_score FROM lead_snapshots WHERE domain=:d ORDER BY last_updated DESC LIMIT 1"),
    {"d": domain}
).fetchone()
Breakingscheduler started at module level — causes duplicate scheduler on uvicorn --reload
In Stage 4's main.py, the scheduler is created and started directly at module import time, outside any lifecycle hook. When running with uvicorn --reload, the module is re-imported on every file change. Each re-import starts a new scheduler instance, so you accumulate duplicate background threads all firing simultaneously. After 5 file edits you have 5 schedulers running.

Fix: Move the scheduler into FastAPI's lifespan context manager:
from contextlib import asynccontextmanager

@asynccontextmanager
async def lifespan(app: FastAPI):
    scheduler.start()
    yield
    scheduler.shutdown(wait=False)

app = FastAPI(lifespan=lifespan)
